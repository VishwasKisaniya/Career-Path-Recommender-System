Technical Deep Dive — Next Skill Recommender System
________________


1. Core Recommendation Engine
Technology: Neo4j Graph Database + Cypher Query Language
The heart of the system is a directed knowledge graph where nodes are Skill entities and edges are UNLOCKS relationships. The fundamental query is:
(In cypher)
MATCH (a:Skill)-[r:UNLOCKS]->(b:Skill)
WHERE a.name IN $skills
RETURN b.name, r.track, r.reason, b.difficulty
Given a set of skills the user knows, the system traverses outgoing edges and returns everything those skills unlock. This is a single-hop graph traversal — intentional, because recommending too far ahead (2-3 hops) would suggest skills the user isn't ready for yet.
Why graph over a relational DB: Prerequisite relationships are naturally graph-structured. SQL would require complex self-joins to express "skills unlocked by skills." Neo4j expresses this natively and efficiently.
________________


2. Skill Node Properties
Every Skill node carries:
* name — canonical standardized name (e.g. "Node.js" not "nodejs")
* difficulty — integer 1 to 5:
Value
	Meaning
	Examples
	1
	Absolute beginner
	HTML, Git basics
	2
	Beginner
	Python, SQL intro
	3
	Intermediate
	Docker, React, REST APIs
	4
	Advanced
	Kubernetes, AWS, System Design
	5
	Expert
	Distributed Systems, MLOps
	Difficulty is set during YAML import for seed data, and by the LLM during dynamic expansion for new nodes.
________________


3. UNLOCKS Relationship Properties
Every UNLOCKS edge carries:
* track — career domain from a fixed enum: ["Web", "Data", "DevOps", "ML", "Mobile", "Systems", "Cloud", "Core", "Automation"]
* reason — human-readable explanation of why this skill unlocks the next
The fixed enum for track is critical — without it, the LLM would generate inconsistent track names like "DevOps", "Dev Ops", "Cloud/DevOps" across different calls, causing the same skill to appear under multiple tracks for different users.
________________


4. Dynamic Graph Expansion (LLM Integration)
Technology: Google Gemini 2.5 Flash via google.generativeai
Two scenarios trigger LLM calls:
Scenario A — New skill not in graph: User types a skill that doesn't exist in Neo4j. The system:
1. Calls validate_skill() — LLM checks if it's a real technical skill and returns its canonical name ("react js" → "React")
2. If valid, calls expand_skill() — LLM generates recommended next skills with track, reason, difficulty
3. All expansions are written to Neo4j, growing the graph permanently
Scenario B — Leaf node (existing skill with no outgoing edges): User types a skill that exists but has no UNLOCKS edges. Same expansion process as above — the LLM fills in the frontier.
LLM output validation before writing to graph: Every item from expand_skill() is checked for:
* skill field is a non-empty string
* track field is a non-empty string
* reason field is present
* difficulty field is an integer
Any malformed item is silently skipped — garbage never enters Neo4j.
Retry logic: Both validate_skill() and expand_skill() retry once with a 1-second delay if the API call fails, before giving up gracefully.
________________


5. Graph Integrity Features
Duplicate Node Prevention
The LLM might generate "Node.js" one session and "NodeJS" another session. Both would become separate nodes in Neo4j representing the same skill.
Fix: Every LLM-generated neighbor skill name is passed through normalize_skill_name() before being written — which calls validate_skill() again to get the canonical form. Neo4j's MERGE then ensures no duplicate node is created even if normalization fails.
Cycle Prevention
The LLM could theoretically generate Python → Django → Python, creating a cycle. Before writing any edge A → B, the system checks:
(In cypher)
MATCH (a:Skill {name: $to})-[:UNLOCKS]->(b:Skill {name: $from})
RETURN COUNT(*) AS cnt
If the reverse edge already exists, the new edge is skipped entirely.
Self-loop Prevention
Simple string equality check before any add_relationship() call — if from_skill == to_skill, return immediately.
Track Normalization
If the LLM returns a track name not in the valid enum (e.g. "Cloud Computing" instead of "Cloud"), it gets normalized to "Core" as a safe fallback.
________________


6. In-Memory Skill Cache
(In python)
_skill_cache: list = []
On startup, all skill names are loaded from Neo4j into this Python list. skill_exists() checks this list instead of querying Neo4j on every request. This avoids N+1 queries when processing a batch of user skills.
The cache is refreshed after every /recommend call that adds new nodes, so new skills become immediately available for future lookups.
________________


7. User Identity Without Login
Technology: crypto.randomUUID() in browser + localStorage
On first visit, the frontend generates a UUID and stores it in localStorage:
(In javascript)
function getOrCreateUserId() {
  let userId = localStorage.getItem("skill_recommender_user_id");
  if (!userId) {
    userId = crypto.randomUUID();
    localStorage.setItem("skill_recommender_user_id", userId);
  }
  return userId;
}
This UUID is passed with every /recommend request as user_id. The backend uses it to identify the user across sessions — no registration, no password, no auth complexity.
Limitation acknowledged: Device and browser specific. Clearing localStorage generates a new identity.
________________


8. Persistent User History
Technology: SQLite via Python's built-in sqlite3
Two tables:
users
user_id | created_at | last_accessed
user_skill_history
id | user_id | skill | chosen_in_session | learned_at
The chosen_in_session flag is the key design decision — it's 1 for skills the user consciously submitted, 0 for LLM-generated neighbors. Only chosen_in_session = 1 skills are used for level calculation. This prevents auto-generated graph expansions from artificially inflating the user's level.
save_skills() also checks for duplicates before inserting — the same skill is never recorded twice for the same user.
________________


9. User Level Calculation
Algorithm in career.py:
Step 1: Fetch difficulty values of all the user's consciously submitted skills from Neo4j (only where difficulty IS NOT NULL)
Step 2: Sort ascending and drop bottom 25%
* Prevents beginner skills from dragging down a senior developer's score
* A developer who knows Kubernetes, AWS and also knows HTML — HTML gets dropped
Step 3: Score remaining skills:
difficulty 1–2  →  2 points
difficulty 3    →  4 points
difficulty 4–5  →  6 points
Step 4: Sum to final score, map to label:
score < 30   → Beginner      → difficulty_level 2
score < 50   → Mid-Level     → difficulty_level 3
score < 70   → Senior        → difficulty_level 4
score ≥ 70   → Architect     → difficulty_level 5
The difficulty_level output directly feeds into recommendation filtering.
________________


10. Level-Aware Recommendation Filtering
Once the user's difficulty_level is calculated, recommendations are filtered:
python
min_diff = max(1, user_diff_level - 1)  # never go more than 1 level below
max_diff = min(5, user_diff_level + 2)  # recommend up to 2 levels ahead
A Senior user (difficulty_level 4) sees only difficulty 3–5 recommendations — never gets Python or HTML suggested. A Beginner sees difficulty 1–3 — never gets Kubernetes suggested prematurely.
The b.difficulty IS NULL guard in the Cypher query ensures newly added skills without difficulty yet still appear rather than being silently excluded.
________________


11. Specialization Detection
(In cypher)
MATCH (a:Skill)-[r:UNLOCKS]->(b:Skill)
WHERE a.name IN $skills
RETURN r.track AS track, COUNT(*) AS cnt
ORDER BY cnt DESC
Counts outgoing edges from the user's skills grouped by track — whichever track has the most connections is the user's specialization.
Tiebreaker: If two tracks are tied, the track of the user's most recently learned skill (from SQLite ORDER BY learned_at) is used as the tiebreaker. This is a timestamp-based resolution — more accurate than random ordering.
________________


12. Already-Known Skills Filter
Before returning recommendations, the system filters out any skill the user already submitted in the current request:
(In python)
filtered = [s for s in skills if s["skill"] not in normalized_inputs_set]
Prevents Python → Django from appearing as a recommendation if the user already told the system they know Django.
________________


13. Input Deduplication and Cap
(In python)
input.skills = list(dict.fromkeys(input.skills))  # ["Python","Python"] → ["Python"]


if len(input.skills) > 10:
    return {"error": "Maximum 10 skills allowed per request."}
Deduplication prevents double LLM API calls for the same skill. The cap prevents API quota abuse.
________________


14. Export and Reset
Export (GET /user/{user_id}/export): Returns full user profile and skill history as JSON. Frontend triggers a browser file download.
Reset (POST /user/{user_id}/reset): Deletes all rows from users and user_skill_history for that user. The Neo4j graph is completely unaffected — shared graph remains intact, only personal history is wiped. Frontend clears localStorage and reloads, generating a fresh UUID.
